## Project overview

**Wenger Field Notes** is a mobile-first sales field app for Wenger Corporation's California sales team (5 reps + 2 admins) covering California private schools. Reps log school visits on their phones; an admin dashboard aggregates submissions; Claude AI generates pipeline summaries from the raw data.

**Tech stack (do not deviate without approval):**

- **Framework:** Next.js 15+ with App Router, TypeScript strict mode
- **Styling:** Tailwind CSS v4, mobile-first responsive
- **UI Components:** shadcn/ui where appropriate, otherwise plain React
- **Database:** Neon Postgres (via Vercel Marketplace), Drizzle ORM
- **Auth:** NextAuth v5 (Auth.js) with Email magic links via Resend
- **AI:** @anthropic-ai/sdk, model `claude-sonnet-4-5`, streaming responses
- **Hosting:** Vercel, auto-deploy from `main` branch
- **Repo:** GitHub, private

- **Primary brand color:** Wenger navy `#0A3758`

- **Inner app header:** Wenger navy `#0A3758` solid background bar, white Wenger logo left-aligned, user controls right-aligned.
- **Typography:** System font stack (`ui-sans-serif, system-ui, -apple-system`) everywhere, including display headings. No serif fonts anywhere in the app.
- **Tone:** Refined, professional, calm. Not playful. Not corporate-stiff either. Think "premium tool a craftsperson uses daily."

## UI/UX principles (non-negotiable)

1. **Mobile-first, always.** Every screen must work flawlessly on a 375px-wide phone before you optimize for desktop. Test in mobile viewport first.
2. **Smooth transitions.** Every navigation, every modal open/close, every state change uses subtle motion (150-250ms ease-out). Use Tailwind transitions or Framer Motion if needed. Never abrupt.
3. **Touch targets ≥ 44px.** Buttons, tabs, list items must be tappable with a thumb.
4. **One primary action per screen.** Reps in the field are distracted. Make the next step obvious.
5. **Loading and empty states are required.** Never leave the user staring at a blank screen. Skeleton loaders, empty-state illustrations, helpful copy.
6. **Error messages are human.** "Couldn't save your visit — check your connection and try again" not "Error 500."
7. **Keep navigation shallow.** Max 2 levels deep. Reps shouldn't get lost.
8. **Accessibility baseline:** semantic HTML, alt text on images, keyboard navigation works, color contrast WCAG AA minimum.

## Cycle discipline (this is the most important section)

This project builds in **cycles**. Each cycle is a small, shippable slice. After every cycle, we:

1. Test locally
2. Commit to Git with a descriptive message
3. Push to GitHub (triggers Vercel auto-deploy)
4. Verify the deployment works on the live URL
5. Update `CHANGELOG.md` with what shipped
6. Update `README.md` if user-facing setup or features changed
7. Update `CLAUDE.md` only if architectural rules changed (requires explicit approval)

**Before starting any new cycle, Claude Code MUST do this checklist (no exceptions):**

```

[ ] Read CHANGELOG.md — confirm what shipped in the previous cycle
```

## Cycle plan (the roadmap)

Cycles are numbered. Each has an objective, a scope, and a done-criteria. Cycles execute in order. Do not skip ahead.

### Cycle 1 — Foundation & branded login (CURRENT)
**Objective:** Get a deployable Next.js app on Vercel with the Wenger-branded login screen rendering.
**Scope:** Next.js scaffold, Tailwind, brand colors as CSS vars, gradient login page with white logo, rep selector UI (no auth yet — just UI). Both logos in `/public`. Deploy to Vercel.
**Done when:** Live URL shows the login screen with gradient background, white Wenger logo, and 6 rep buttons. Selecting a rep shows a placeholder "next phase" screen. No database, no auth, no form yet.

### Cycle 2 — Branded app shell & navigation
**Objective:** Build the in-app shell that wraps every authenticated screen.
**Scope:** Persistent Wenger navy header with white logo + user pill + logout. Tab bar for Map / My Submissions. Placeholder pages for each tab. Smooth route transitions.
**Done when:** After selecting a rep on login, user lands in app shell. Can navigate between tabs. Logo and brand styling consistent throughout.

### Cycle 3 — School data & interactive map
**Objective:** Bring the 47 schools into the app and render them on an interactive map.
**Scope:** `lib/schools.ts` data array. Map view with pins (use Leaflet + OpenStreetMap free tiles, OR canvas-based pins if Leaflet adds too much). Search/filter UI. Tap a pin → school detail preview.
**Done when:** Map shows all 47 schools. Search filters list. Tapping a school opens a preview card with "Start visit" button.

### Cycle 4 — Visit form
**Objective:** Build the full visit form with the priority block at top.
**Scope:** Form layout matching prototype (priority block, contact, purchasing, decision-making, marketing, notes sections). Multi-select checkboxes. Auto-save to local state. "Save submission" button (still local-only, no DB yet).
**Done when:** Reps can fill out the form and see it appear in a local "My Submissions" list. Same fields as the prototype. Priority block visually distinct.

### Cycle 5 — Database & persistence
**Objective:** Wire Neon Postgres so submissions persist server-side.
**Scope:** Neon integration via Vercel Marketplace, Drizzle ORM schema, migrations, API routes for create/read submissions. Replace local state with real DB calls.
**Done when:** Submissions saved on one device appear on another. Data survives browser refresh.

### Cycle 6 — Real authentication
**Objective:** Replace the rep selector with real authentication. Users log in with email + password.
**Scope:** NextAuth v5 with Credentials provider, JWT sessions, `users` table in Neon with bcrypt-hashed passwords seeded from `lib/allowlist.ts` using a generic bootstrap password (`SEED_PASSWORD` env var), force-change-on-first-login flow, session handling, protected routes via middleware, server-side scoping of `/api/submissions` (reps see own; admins see all). No email sending; no Resend.
**Done when:** Only allowed users can log in with email + password. First login forces password change before granting full session. Each rep sees only their own submissions. Admins see all. The rep selector is gone.

### Cycle 6.5 — User self-service profile edit
**Objective:** Logged-in users can edit their own profile.
**Scope:** `/account` page with form to update name, email, and password (current-password confirmation required for password change). One API route. No admin-facing UI.
**Done when:** Any logged-in user can change their own name, email, and password. Changes persist in Neon and reflect in the session immediately.

### Cycle 7 — Admin dashboard
**Objective:** Build the admin view: submissions list, filters, CSV export, and full user management.
**Scope:** `/admin` route, gated to admin users. Submissions: stats row, filter by rep, expandable submission detail, CSV export. User management: list users, add user, edit user (name/email/role/repId), reset user password (admin override; resets `password_updated_at` so target user is forced to change on next login), remove user with mandatory reassignment step ("This user has N submissions — reassign to whom?" dropdown of other users, executed in a transaction).
**Done when:** Admin can view, filter, and export all team submissions. Admin can add, edit, and remove users. Deleting a user with existing submissions requires choosing a reassignment target; the operation is atomic.

### Cycle 8 — Claude AI summaries
**Objective:** Wire the Anthropic API for pipeline summaries.
**Scope:** `/api/summarize` route, streaming Claude responses, "Summarize pipeline" and "Per-rep summary" buttons on admin dashboard, loading states.
**Done when:** Admin clicks a button and sees Claude analyze real submissions and stream back a useful summary.

### Cycle 9 — Polish & launch
**Objective:** Final QA, copy review, custom domain, team onboarding.
**Scope:** Loading skeletons, error boundaries, empty states, custom domain setup, team onboarding (handoff to admin Mark for distribution; no email sending — app has no email infrastructure).
**Done when:** Team is using it. Real data is flowing.

### Cycle 10 — Edit & delete submissions
**Objective:** Reps can edit and delete their own submissions; admins can edit and delete any submission.
**Scope:** `PATCH` + `DELETE` on a single `/api/submissions/[id]` route (`runtime=nodejs`), owner-or-admin enforced server-side like the existing GET (401/404/403); PATCH body-validated the same way `POST /api/submissions` is; identity (`id`/`repId`/`repName`/`schoolId`/`schoolName`) is immutable so an admin edit never reassigns ownership; `updatedAt` bumped. Reuse the existing visit form, prefilled, in an edit mode that bypasses the localStorage draft system. Delete uses the existing `useToast().confirm()` modal (no `window.confirm`). Entry points: `/submissions` list + `/submissions/[id]` detail (rep, own rows) and the `/admin` Submissions expandable detail (admin, any row). New `/submissions/[id]/edit` route gets `loading.tsx`/`error.tsx`; reuse `<Button>`/`useToast`/skeletons/`<ErrorState>`.
**Done when:** A rep can edit and delete only their own submissions and an admin any (server-enforced); edits persist in Neon and reflect in list/detail/admin; an admin edit never reassigns the submission's rep; deletes use the toast confirm modal; build clean; verified live.

### Cycle 11 — Standalone/PWA presentation polish
**Objective:** The installed/standalone PWA feels native (no overlap, nothing offscreen, no resize glitches) and the app header is layout-aware.
**Scope:** Web app manifest (standalone display, `start_url`, `scope`, `theme-color = #0A3758`, icons), `viewport-fit=cover`, and safe-area-inset handling so the navy header and the Map / My Submissions tab bar never collide with the OS status bar or home indicator. Adaptive header: on full-width pages the logo sits flush far-left and the user/nav controls flush far-right; on boxed/constrained pages keep the current content-width alignment — driven by a layout-aware signal, not per-page hardcoding. Test at 375px and installed standalone (iOS + Android). Flag any new dependency before adding it.
**Done when:** Installed to the home screen on iOS and Android the app shows no overlap/offscreen/resize glitches; header alignment adapts correctly to full-width vs boxed routes; 375px is unaffected; build clean; verified live.

### Cycle 12 — Offline-first foundation
**Objective:** Field reps can work through dead zones without losing data — drafts survive, submissions queue locally, pending submissions remain editable until they sync, and the sync drains automatically when connectivity returns.
**Scope:** Service worker via `@serwist/next` precaching app shell + JS/CSS + logos + `lib/schools.ts` (builds on Cycle 11's manifest; no manifest changes). Dexie-backed IndexedDB with two stores: `drafts` (in-progress forms, keyed by `repId + schoolId`, ~500ms debounced autosave from the visit form, cleared on successful submission) and `pending_submissions` (`status: pending|syncing|synced|failed`, `schema_version` for future migrations). Submission flow rewritten local-first: "Save submission" writes to `pending_submissions` and shows immediately in My Submissions with a "Pending sync" badge, then attempts `POST /api/submissions` (success → remove local copy; failure → stays pending for retry). Pending submissions are editable/deletable offline through the same Cycle 10 UI — taps from `/submissions` or detail route to the edit form, which writes back to IndexedDB instead of `PATCH /api/submissions/[id]`; identity (`repId`/`schoolId`/…) immutable. Mid-edit race (row transitions to `syncing`/`synced`): the save handler detects the status change and redirects to the online edit flow with a brief notice. Synced submissions remain online-only edit/delete (Cycle 10 unchanged). Sync triggers: app load, browser `online` event, manual "Sync now" button, and a 60s interval while app is open + online. Connectivity check is a real HEAD ping to `/api/health` (returns `{ok:true,ts}`), not `navigator.onLine` (lies on iOS). Sync UI lives on the Submissions tab only — status strip ("✓ All synced" calm gray / "⏳ N pending sync" warm `#b8612a` + "Sync now"), per-row sync badges, numeric badge on the Submissions tab itself visible from any screen; header untouched. OSM map tiles cached via the SW for the California bounding box at zoom 8–14, cache-first with SWR. Auth, admin dashboard, and Claude AI summaries remain online-only by design.
**Done when:** App installs and opens in airplane mode with cached tiles + school list. Filling a form offline saves the draft; closing and reopening restores it. Submitting offline shows the row immediately with "Pending sync"; the same UI lets reps edit/delete it before it syncs (delete never reaches the server). A row that syncs mid-edit redirects cleanly to online edit. Restoring connectivity drains the queue within seconds; rows flip to synced. A row saved offline on one device appears on another after sync. Force-quit + reopen offline preserves drafts and pending. Status strip and tab badge accurately reflect queue state from anywhere in the app. "Sync now" works as a safety valve. Build clean; verified live; tested with DevTools offline throttling **and** real airplane-mode on the installed PWA.
**New deps (flag in cycle summary):** `@serwist/next`, `serwist`, `dexie`, `dexie-react-hooks`. **New files:** `/api/health`, `src/lib/db/local.ts`, SW wrapper in `next.config.ts`, `public/sw.ts` (or serwist entry equivalent). **Architectural notes:** server remains source of truth (local DB = write-ahead cache + draft store); no conflict-resolution logic (synced rows are append-only from the client; offline edits only apply to records the server hasn't seen); offline edit of synced records deferred indefinitely; `schema_version` is groundwork for future shape migrations; session cookie carries the rep through offline use after online login.
